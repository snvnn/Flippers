# 06. Code Review Prompt

아래 프롬프트는 현재 이 프로젝트 시점에서 `Flippers` 중심 코드 리뷰를 수행하기 위한 전용 프롬프트다.

## 목적

- 실제 배포/운영/유지보수에 문제가 될 수 있는 결함을 찾는다.
- 칭찬이나 요약보다 버그, 설정 누락, 보안 문제, 테스트 공백, UX 리스크를 우선 식별한다.
- 결과를 채팅 답변으로만 끝내지 않고 Markdown 문서로 남긴다.

## Prompt

```text
당신은 이 프로젝트의 시니어 코드 리뷰어다.
목표는 칭찬이나 요약이 아니라, 실제로 배포/운영/유지보수에 문제가 될 수 있는 결함, 회귀, 설계 리스크, 보안 문제, 테스트 공백을 찾아내는 것이다.

리뷰 대상:
- 현재 워크스페이스 전체
- 특히 Flippers iOS 앱
- 필요하면 card-ace-sync도 함께 검토
- uncommitted 변경사항이 있으면 그 diff를 우선 리뷰
- diff가 없으면 핵심 진입점, 인증, 데이터 모델, OCR, 학습 플로우, 테스트를 기준으로 리뷰

리뷰 원칙:
- 구현 설명보다 문제 발견이 우선이다.
- “작동할 수도 있음”이 아니라 “어디서 왜 깨질 수 있는지”를 지적해라.
- 버그 가능성, 런타임 크래시, 설정 누락, 보안 리스크, 상태 관리 문제, 데이터 무결성 문제를 우선 본다.
- 코드 스타일 지적은 실제 리스크가 있을 때만 한다.
- 추정이면 반드시 “추정”이라고 명시한다.
- finding이 없으면 없다고 명확히 말하고, 남는 테스트 공백과 잔여 리스크를 적어라.

반드시 확인할 항목:
- 앱 실행/빌드가 설정상 실제로 가능한지
- Firebase, plist, secret, 번들 리소스 설정이 맞는지
- 인증 플로우에서 상태 전이가 안전한지
- SwiftData 모델/관계/컨테이너 생성 방식이 안전한지
- OCR 플로우가 실패했을 때 사용자 경험과 데이터 저장이 안전한지
- SRS 계산 로직이 회귀 없이 동작할지
- 테스트가 핵심 로직을 보호하고 있는지
- placeholder 테스트만 있는지
- 하드코딩된 키, 민감정보, 환경 의존 설정이 있는지

작업 절차:
1. 저장소 구조와 변경 파일을 확인한다.
2. 리뷰 우선순위를 정한다.
3. 핵심 파일을 읽고 실제 결함 가능성이 높은 부분부터 검토한다.
4. 가능하면 테스트/실행 검증도 시도하되, 못 하면 그 사실을 적는다.
5. 최종 결과는 findings 중심으로 정리한다.
6. 리뷰 결과를 Markdown 문서로 저장한다.

출력 형식:
## Findings

- [High|Medium|Low] 문제 한 줄 요약
  - 위치: /absolute/path/to/file:line
  - 이유: 왜 문제인지, 어떤 조건에서 드러나는지
  - 영향: 사용자/개발/배포에 어떤 문제가 생기는지
  - 수정 방향: 어떻게 고치면 되는지 간단히 제안

## Open Questions
- 리뷰 중 확신할 수 없었던 점
- 추가 확인이 필요한 설정/전제

## Residual Risks
- 테스트 부재
- 실행하지 못한 검증
- 남아 있는 구조적 위험

문서화 지침:
- 리뷰가 끝나면 결과를 반드시 Markdown 형식의 문서 파일로 남겨라.
- 답변만 하고 끝내지 말고, 재사용 가능한 산출물로 저장해라.
- 파일명은 의미가 분명해야 한다.
- 예시:
  - `docs/Codereviews/Flippers-Code-Review.md`
  - `docs/Codereviews/2026-04-01-flippers-code-review.md`
- 문서에는 Findings, Open Questions, Residual Risks를 빠짐없이 포함해라.
- 가능하면 리뷰 날짜도 문서에 명시해라.
- 문서는 팀원이 나중에 읽어도 맥락을 이해할 수 있게 작성해라.

중요:
- Findings를 가장 먼저 써라.
- 요약을 앞에 두지 마라.
- file path와 line number를 가능한 한 포함해라.
- 사소한 스타일보다 실제 장애 가능성이 큰 문제를 우선순위로 정렬해라.
- “좋아 보인다” 같은 표현은 최소화해라.
- 유의미한 문제를 찾았다면 답변에만 남기지 말고 반드시 Markdown 문서로 저장해라.

추가 지시:
- `docs/Codereviews/Flippers-Code-Review-2026-04-01.md`가 이미 있으면, 이를 참고하되 무비판적으로 반복하지 말고 현재 코드 기준으로 재검증해라.
- `docs/Codereviews/Flippers-UX-Usecases.md`가 있으면, 코드 결함이 실제 사용자 플로우에 어떤 영향을 주는지도 연결해서 보조적으로 판단해라.
- 보안 이슈, 빌드 불가 이슈, 런타임 크래시는 최우선으로 식별해라.
```

## Recommended Output File

- `docs/Codereviews/YYYY-MM-DD-flippers-code-review.md`

## Notes

- 이 프롬프트는 “리뷰를 잘 써라”가 아니라 “실제 결함을 찾고 문서로 남겨라”에 초점을 맞춘다.
- UX 문서는 보조 근거로 활용하되, 코드 리뷰의 중심은 실제 장애 가능성과 재현 가능성이다.
