> 분석 기준일: 2026-04-01
> 분석 방법: 코드베이스 직접 열람 + 사용자 시뮬레이션

## 제품 한 줄 정의

**OCR로 교재를 찍어 카드를 만들고, SRS 알고리즘으로 일본어/한자를 반복 학습하는 iOS 플래시카드 앱**

---

## 핵심 사용자 유형

### U1 — 자습생 (핵심 사용자)
- 일본어 교재·단어장을 가지고 혼자 공부하는 사람
- 디지털 도구에 익숙하며 Anki 같은 SRS 앱을 써봤을 수도 있음
- 목표: 교재 내용을 빠르게 디지털화해서 이동 중에도 학습하고 싶다

### U2 — 시험 준비생
- JLPT·한자검정 등 특정 시험을 앞두고 단기 집중 학습
- 앱을 처음 쓰는 사람; "복잡하면 그냥 안 씀"
- 목표: 시험 범위 단어를 빠르게 등록하고 매일 복습하고 싶다

### U3 — 취미 학습자
- 드라마·만화·게임에서 본 단어를 저장해두는 사람
- 불규칙하게 사용; 긴 공백 후 재진입하는 패턴
- 목표: 흥미로운 단어를 잊기 전에 저장하고 나중에 복습하고 싶다

## 유스케이스별 상세 시뮬레이션

---

### UC1 — 첫 실행 · 회원가입

- **사용자 목표**: 앱을 설치하고 바로 학습을 시작하고 싶다
- **진입 조건**: 앱 최초 실행, 계정 없음
- **기대하는 결과**: 계정을 만들고 학습 화면에 진입한다

**실제 이용 플로우:**
1. 앱 실행 → `ContentView`가 `authViewModel.isLoggedIn`을 확인
2. 미로그인 → `AuthView` 표시
3. "앱 이름 Flippers + 부제 일본어 플래시카드" 헤더 확인
4. 이메일·비밀번호 입력 → 기본 모드는 **로그인** 모드
5. 회원가입하려면 하단 "계정이 없으신가요? 회원가입" 텍스트 링크 탭
6. 모드 전환 후 다시 이메일·비밀번호 입력
7. "회원가입" 버튼 탭 → Firebase Auth 호출 → `isLoading = true` → ProgressView
8. 성공 → `currentUserUID` 설정 → `MainTabView` 진입
9. 탭 4개: 학습 / 카드 / 개요 / OCR

**사용자 감정 변화:**
> 설치 직후 기대감 → 회원가입 모드를 못 찾아 혼란 → 모드 전환 후 안도 → 로그인 성공 후 진입 → 카드가 하나도 없어서 어디서 시작해야 할지 막막

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | 앱을 처음 켰을 때 기본값이 "로그인" 모드다. 처음 쓰는 사람은 회원가입 버튼을 찾아야 한다. Anki·Duolingo 등 경쟁 앱은 온보딩을 먼저 보여준다. | **High** |
| B | `MainTabView` 진입 직후 학습 탭 기본값인데, 카드가 없으면 "오늘 복습 없음 + 새 카드를 추가하거나 내일 다시 오세요"가 뜬다. "내일 다시 오세요"라는 문구는 신규 사용자에게 최악의 첫인상이다. | **High** |
| C | 회원가입 → 로그인 전환 링크가 본문 하단에 작은 텍스트로만 있다. 44pt 탭 영역 기준에 못 미칠 가능성 **(추정)**. | **Medium** |
| D | Apple 로그인 버튼이 이메일 폼 아래에 있어 스크롤해야 보일 수 있다 **(추정, 화면 크기 의존)**. | **Low** |

**실패 시나리오:**
- Firebase 네트워크 오류 → `errorMessage` 표시 (Firebase 원문 영어 메시지가 그대로 노출됨 **(추정)**)
- 비밀번호 6자 미만 → Firebase 오류 → 사용자에게 직접적인 가이드 없이 영문 오류 메시지

**개선 아이디어:**
1. 앱 최초 실행 시 3장짜리 온보딩 슬라이드 또는 "게스트로 시작" 옵션 제공
2. 회원가입/로그인 탭 UI로 변경 (Picker segmented 또는 탭 헤더)
3. `MainTabView` 진입 후 카드가 0개면 "첫 카드 만들기" CTA를 학습 탭에 직접 노출
4. Firebase 오류 메시지를 한국어로 변환하는 래퍼 추가

### UC2 — OCR로 카드 일괄 등록

- **사용자 목표**: 교재 단어 목록을 찍어 한 번에 카드로 만들고 싶다
- **진입 조건**: 로그인 완료, 단어장 이미지 보유
- **기대하는 결과**: 이미지에서 단어·읽기·뜻을 뽑아 카드가 생성된다

**실제 이용 플로우:**
1. OCR 탭 탭
2. "사진 라이브러리에서 선택" 또는 "카메라로 촬영" 선택
3. 이미지 선택 → `processImage()` 호출 → "텍스트 추출 중…" 스캔 화면
4. Vision on-device 처리 → Claude API 보완 (네트워크 필요)
5. 리뷰 화면: 추출된 단어 목록, 각 행에 체크박스·한자·읽기·뜻
6. 불필요한 단어 체크 해제 → "N개 카드 생성" 버튼 탭
7. Done 화면: "カード作成完了！" + 생성 개수 + "추가 스캔" 버튼

**사용자 감정 변화:**
> OCR 기능에 기대감 → 스캔 중 대기 (얼마나 걸릴지 모름) → 결과 화면에서 추출 품질에 따라 만족/실망 → 생성 후 어느 덱에 들어갔는지 확인 불명확 → 학습 탭으로 가도 카드가 안 보이면 혼란

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | OCR 추출이 **x좌표 비율 기반 column 분리** 방식이다. 교재 레이아웃이 "15–35% 구간에 한자, 35–55%에 읽기"라는 가정에 의존한다. 일반 교재는 이 레이아웃을 따르지 않는 경우가 많다 — 전부 "kanji" 열에 몰리거나, 읽기·뜻이 비어있는 결과가 나올 수 있다. | **High** |
| B | 스캔 화면에 진행률(%) 또는 소요 시간 예상 없이 무한 spinner만 표시된다. Claude API 호출이 포함되므로 5–15초 이상 걸릴 수 있다. 사용자는 앱이 멈춘 건지 모른다. | **High** |
| C | OCR 업로드 단계에서 덱 선택이 있는데, 덱이 없으면 어떻게 되는지 UI가 불명확하다. `selectedDeck = nil` 상태로 카드를 생성하면 덱 없는 고아 카드가 된다. 나중에 학습 탭에서 이 카드들이 dueCards에 포함은 되지만, 어느 덱인지 알 수 없다. | **High** |
| D | 리뷰 화면에서 단어 편집이 불가능하다 **(추정: OCRWordRow에 TextField가 없음)**. 추출이 잘못됐어도 수정 없이 저장하거나 전부 체크 해제해야 한다. | **Medium** |
| E | "추가 스캔" 버튼만 있고, "생성된 카드 보러가기" 이동 동선이 없다. 카드가 어디 들어갔는지 확인하려면 탭을 직접 전환해야 한다. | **Medium** |
| F | Claude API 키가 `Config.plist`에서 로드된다. 키가 없거나 만료되면 enhanceWords가 실패하고 Vision 결과만 사용한다 — 이 fallback은 적절하지만 사용자에게 "Claude 보완 없이 저장됩니다"라는 안내가 없다. | **Low** |

**실패 시나리오:**
- 일반 교재(세로 쓰기, 한 줄에 단어만) 촬영 → 한자만 추출, 읽기/뜻 전부 비어있음 → 카드 생성은 되지만 학습에 쓸 수 없는 카드
- 오프라인 → Vision 성공, Claude 실패 → fallback 동작하지만 읽기/뜻 없이 저장
- 이미지 내 텍스트가 없거나 해상도 낮음 → `extractedWords` 비어있음 → ContentUnavailableView "단어 없음"만 표시, 재촬영 유도 없음

**개선 아이디어:**
1. 스캔 단계에 단계별 진행 메시지 표시 ("텍스트 인식 중… / 읽기 보완 중…")
2. 리뷰 화면에서 행 탭 시 인라인 편집 기능 추가
3. OCR 시작 전 "이 앱은 한 줄에 한자 / 읽기 / 뜻이 구분된 레이아웃에 최적화됩니다" 안내 또는 레이아웃 유형 선택 옵션
4. Done 화면에서 "카드 탭에서 확인" 딥링크 버튼 추가
5. 덱 미선택 상태 명시적 경고 또는 자동 임시 덱 생성

### UC3 — 수동으로 카드 한 장 만들기

- **사용자 목표**: 드라마에서 본 단어 하나를 빠르게 저장하고 싶다
- **진입 조건**: 로그인 완료, 단어 하나 기억
- **기대하는 결과**: 30초 안에 카드 한 장 저장

**실제 이용 플로우:**
1. 카드 탭 → 우상단 "+" 버튼
2. `CardEditView` 시트 열림
3. 유형 선택 (word/kanji segmented picker)
4. 필드 입력: 단어, 읽기, 뜻, 예문 (순서대로 4개 TextField)
5. 덱 선택 Picker
6. "저장" 버튼 → `createCard()` → 카드 + 필드 + SRSState insert
7. 시트 닫힘 → 카드 목록에 즉시 표시

**사용자 감정 변화:**
> 단순해서 빠르게 시작 → 덱이 없으면 "새 덱 만들기" 버튼이 보여 흐름이 끊김 → 덱 만들고 돌아와 저장 → "잘 저장됐나?" 확인하려면 목록으로 눈이 감

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | 덱이 없을 때 CardEditView 내에서 덱을 새로 만들 수 있지만, 만들고 나면 해당 덱이 자동 선택되는지 불명확하다 **(추정: 코드 상 `showAddDeck` sheet가 닫힌 후 Picker 재갱신 여부)**. 수동으로 다시 선택해야 한다면 불편. | **Medium** |
| B | 카드 저장 후 성공 피드백(토스트·햅틱 등)이 없다. 시트가 그냥 닫힌다. 저장됐는지 순간적으로 불안. | **Medium** |
| C | word 타입에 "예문" 필드가 있지만 placeholder나 가이드가 없다. 일본어 예문을 직접 입력해야 하는데, 기존 예문을 어디서 가져와야 하는지 앱이 안내하지 않는다. | **Low** |
| D | 한자 타입의 "음독/훈독" 필드명이 한국어 사용자에게 익숙하지 않을 수 있다. 音読み / 訓読み 표기와 한국어 레이블이 혼재 **(추정)**. | **Low** |

**실패 시나리오:**
- 단어 필드(primary field) 비워두고 저장 시도 → 버튼 disabled, 안내 없음
- 덱 없이 저장 → 고아 카드 생성, 나중에 덱 연결 방법 없음

**개선 아이디어:**
1. 저장 성공 후 햅틱 피드백 + "카드 저장됨" 토스트 (0.5초)
2. 덱 없을 때 "기본 덱" 자동 생성 또는 덱 없이도 저장 가능하도록 허용
3. 예문 필드에 "예: 彼は学校に行く" 형태의 placeholder 추가

### UC4 — 학습 세션 시작 및 완료

- **사용자 목표**: 오늘 복습해야 할 카드를 빠짐없이 복습하고 싶다
- **진입 조건**: 로그인 완료, 복습 대상 카드 존재
- **기대하는 결과**: 모든 due 카드를 평가하고 "복습 완료" 화면을 본다

**실제 이용 플로우:**
1. 학습 탭 진입 → `.onAppear` → `session.start(cards: dueCards)`
2. 프로그레스 바 + "残り N枚" 표시
3. 카드 앞면 표시 (한자 or 단어 64pt 폰트)
4. "탭하여 뒤집기" 힌트 텍스트 확인 후 탭
5. 3D flip 애니메이션 → 뒷면: 읽기 + 뜻 + 예문 표시
6. 하단에서 Again / Hard / Good / Easy 버튼 4개 슬라이드업
7. 평가 버튼 탭 또는 스와이프 → 다음 카드
8. 모든 카드 완료 → SessionCompleteView: "お疲れ様！" + 평가 분포 + 다시 시작 버튼

**사용자 감정 변화:**
> 탭하면 바로 학습 시작되어 자연스러움 → 카드 뒤집기 애니메이션이 매력적 → 평가 버튼 4개가 처음엔 어떤 의미인지 설명 없음 → 몇 번 쓰면 익숙해짐 → Again 카드가 큐 뒤로 가는 걸 알게 되면 "언제 끝나지?" 불안감

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | Again / Hard / Good / Easy 의미 설명이 없다. 처음 쓰는 사람은 "Good과 Easy의 차이가 뭔지", "Hard를 누르면 어떻게 되는지" 모른다. 일본어 서브레이블(難しい 등)은 도움이 되지만 SRS 개념 자체 안내는 없다. | **High** |
| B | Again을 누르면 카드가 세션 큐 뒤로 간다. 진행 바는 천천히 채워지는데 "残り N枚"가 줄지 않을 수 있다 — 사용자 입장에서 "버튼 눌렀는데 왜 카드가 안 끝나지?" 혼란. `totalCount = completedCount + queue.count` 구조에서 again 카드는 queue에 남아 totalCount가 초기 dueCards.count보다 더 커질 수 없지만, 직관적으로 "이 카드 또 나왔어?" 반응이 올 수 있다. | **Medium** |
| C | 스와이프 제스처가 "Good"과 "Again"만 매핑됐다. Hard와 Easy는 버튼만으로 가능하다. 제스처-버튼 일관성이 없어 제스처를 배운 뒤에도 결국 버튼을 써야 하는 경우가 생긴다. | **Medium** |
| D | "길게 탭하면 발음 힌트"라는 CLAUDE.md 명세가 있지만, 실제 구현은 별도 "길게 눌러 발음" 버튼이다. Long press gesture가 카드 자체에 없다. 사용자가 카드를 길게 눌러보면 아무 반응이 없어 UI가 설명한 것과 다르다는 느낌. | **Medium** |
| E | 세션 완료 화면에 총 복습 시간, 정확도(%), 연속 학습 스트릭 등 동기부여 정보가 없다. "お疲れ様！" 텍스트와 평가 분포만 있다. | **Low** |
| F | 덱 필터 변경 시 `session.start(cards: dueCards)`가 호출되는데, 현재 학습 중에 필터를 바꾸면 진행 중인 세션이 초기화된다. 실수로 필터를 건드리면 학습이 날아간다. | **Medium** |

**실패 시나리오:**
- 카드에 `srsState`가 nil인 경우 → `rate()` 내 `guard let state = card.srsState` 실패, 조용히 무시됨. 해당 카드가 영구적으로 학습 불가 상태로 남음.
- 학습 도중 앱을 백그라운드로 보냈다가 30분 후 복귀 → `onAppear` 재호출되지 않으면 세션 큐 유지. `onAppear` 재호출되면 큐 재초기화 **(추정: SwiftUI onAppear 동작에 따라 다름)**.
- learning 단계 카드 again → 1분 뒤 재출제 예정인데, 사용자가 60초 안에 다른 카드를 모두 완료하면 다시 해당 카드가 최상단으로 나타남. 이 동작에 대한 안내 없음.

**개선 아이디어:**
1. 첫 학습 세션 시 "평가 가이드" 한 장 오버레이 (Again = 다시 / Hard = 힘들었음 / Good = 맞음 / Easy = 쉬웠음 + 각 결과 설명)
2. 세션 완료 화면에 "정확도 N%" 및 "연속 N일 학습 중" 스트릭 추가
3. 덱 필터 변경 시 학습 중이면 "세션을 초기화할까요?" 확인 알럿
4. 스와이프 제스처에 Hard(위) / Easy(아래) 추가하거나, 스와이프는 Good/Again으로 제한하되 명확하게 안내

### UC5 — 덱/섹션 구조 관리

- **사용자 목표**: 교재 챕터별로 카드를 구분하고 싶다
- **진입 조건**: 카드가 이미 있고, 구조 정리가 필요
- **기대하는 결과**: 덱·섹션 계층으로 카드가 분류된다

**실제 이용 플로우:**
1. 개요 탭 → 덱 목록 → 덱 탭 → `DeckDetailView`
2. 우상단 메뉴 (⋯) → 섹션 추가
3. 섹션 이름 입력 → 저장
4. 카드 탭 → 카드 탭 → `CardEditView` → 섹션 선택

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | 덱 생성 진입점이 2곳이다: 개요 탭 상단 "+" 버튼과 CardEditView 내 "새 덱 만들기" 버튼. 앱에서 가장 기본적인 행동인 "덱 만들기"를 어디서 해야 하는지 처음엔 명확하지 않다. | **Medium** |
| B | 섹션 없는 카드와 섹션 있는 카드가 같은 덱에 혼재할 수 있다. "섹션 없음" 그룹이 DeckDetailView 최상단에 항상 떠 있어, 정리된 덱도 지저분해 보인다. | **Low** |
| C | 카드를 다른 덱·섹션으로 이동하는 기능이 없다. 잘못된 덱에 만든 카드를 옮기려면 삭제 후 재생성해야 한다. | **Medium** |

**실패 시나리오:**
- 덱 삭제 시 `cascade` 삭제로 모든 카드·SRSState·ReviewLog가 함께 삭제된다. 확인 다이얼로그는 있지만 "포함된 카드 N장도 삭제됩니다" 수량 정보가 없다 **(추정: confirmationDialog에 count 표시 여부 불확실)**.

**개선 아이디어:**
1. 덱 삭제 확인 다이얼로그에 "포함된 카드 N장이 함께 삭제됩니다" 문구 추가
2. 카드 이동 기능 (덱/섹션 변경)
3. 개요 탭에서 덱 추가 버튼을 더 prominent하게

### UC6 — 오랜 공백 후 앱 재진입

- **사용자 목표**: 2주 만에 앱을 열었을 때 밀린 복습을 한다
- **진입 조건**: 기존 카드 다수 보유, 장기 미접속
- **기대하는 결과**: 밀린 복습 카드를 확인하고 학습한다

**실제 이용 플로우:**
1. 앱 실행 → 이미 로그인 상태면 바로 `MainTabView`
2. 학습 탭 자동 진입 → dueCards 수십 ~ 수백 장
3. 프로그레스 바 시작 → Again이 많이 발생 → 세션이 길어짐
4. 개요 탭에서 "평균 Ease" 수치가 낮아진 것 확인

**혼란/마찰 포인트:**

| # | 포인트 | severity |
|---|--------|----------|
| A | 재진입 시 "오늘 할 카드: 137장" 같은 요약 없이 바로 학습이 시작된다. 부담감으로 인한 이탈 가능. | **Medium** |
| B | CloudKit 동기화 상태가 UI에 전혀 표시되지 않는다. 다른 기기에서 추가한 카드가 아직 동기화되지 않아도 사용자가 알 수 없다. | **Low** |

**개선 아이디어:**
1. 학습 탭 진입 시 "오늘 복습 N장" 요약 카드 + "시작하기" 버튼 (세션 자동 시작 대신 의도적 시작)
2. 100장 이상이면 "오늘 X장만 학습하기" 세션 분할 옵션

## 공통 UX 리스크

### R1 — 신규 사용자 빈 상태 이탈 (High)
앱 첫 실행 → 로그인 → 학습 탭 기본 진입 → "오늘 복습 없음, 내일 다시 오세요". 카드 하나도 없는 사용자에게 이 메시지는 **이탈 트리거**다. "카드 만들기" 또는 "OCR로 시작하기" CTA가 이 화면에 없다.

### R2 — OCR 레이아웃 의존성 (High)
OCR 추출 품질이 이미지 레이아웃에 크게 의존한다. 앱이 가정하는 "한자/읽기/뜻 3열 표" 형식을 따르지 않는 대부분의 교재에서 추출 결과가 불완전하다. 이 앱의 핵심 차별점인 OCR이 기대와 다른 결과를 내면 사용자 신뢰가 깨진다.

### R3 — SRS 평가 개념 미안내 (Medium)
Again / Hard / Good / Easy 버튼이 각각 무엇을 의미하고, 누르면 다음에 언제 다시 나오는지 설명이 없다. Anki 경험이 없는 사용자에게는 블랙박스다.

### R4 — 에러 메시지 영어 노출 (Medium)
Firebase 인증 오류 메시지가 영어 원문 그대로 사용자에게 노출될 가능성이 높다 **(추정)**. "The password is invalid or the user does not have a password" 같은 메시지를 한국어 앱에서 보면 신뢰도가 떨어진다.

### R5 — 세션 중 덱 필터 변경 시 데이터 손실 (Medium)
학습 세션 중 덱 필터를 바꾸면 `session.start(cards:)`가 재호출되어 현재 세션이 날아간다. 의도치 않은 필터 터치 하나로 진행 중인 학습이 리셋된다.

## 요약

Flippers는 **기능적으로 동작하는 MVP**이지만, 신규 사용자가 앱을 열고 첫 학습을 완료하기까지의 경로에 세 가지 주요 장벽이 있다.

1. **진입 장벽**: 회원가입 후 즉시 빈 화면 → 어디서 시작해야 하는지 모름
2. **핵심 기능 신뢰성**: OCR 추출이 레이아웃 의존적 → 기대와 다른 결과
3. **학습 개념 안내 부재**: SRS 평가 버튼의 의미를 알아야 제대로 쓸 수 있음

이 세 가지를 해결하면 **핵심 사용자 여정(설치 → 첫 카드 등록 → 첫 학습 완료)이 이탈 없이 완주 가능한 수준**이 된다.
