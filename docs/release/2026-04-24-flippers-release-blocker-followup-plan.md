# Flippers Release Blocker Follow-up Plan

작성일: 2026-04-24
상태: 실행용 후속 계획

## 목적

이 문서는 `docs/release/2026-04-21-flippers-release-readiness-checklist.md` 에서 확인된 남은 차단 항목을 실제 작업 순서로 바꾸기 위한 후속 실행 계획이다.

## 수정/보완된 문제점

이번 정리에서 체크리스트는 다음 문제를 보완했다.

- OCR 외부 전송의 **사용자 opt-in / 선택 동의**를 명시했다.
- 운영 로그 보관 정책과 접근 범위를 차단 항목으로 올렸다.
- 사고 대응 / 키 유출 대응 절차를 차단 항목으로 올렸다.
- 이용약관 / 웹사이트 / 지원 링크 항목의 문구를 더 명확하게 바꿨다.

## 아직 해결되지 않은 문제

아래 항목은 여전히 `must-block-release` 이며, 현재 시점에 완료되지 않았다.

1. 공개용 문의 채널 확정
2. 개인정보 처리 문의 / 권리 요청 접수 경로 확정
3. 실제 운영 프록시 배포 대상 확정 및 반영
4. 서버 측 비밀 키 관리 방식 확정
5. 과거 노출 가능 키 폐기 / 교체 여부 확인
6. 개인정보처리방침에 실제 운영 정보 반영 완료
7. App Store Privacy 제출값 확정
8. 공개 문구와 실제 구현 / 운영 구조 정합성 최종 점검
9. 계정 삭제 요청 경로 확정
10. 전체 데이터 삭제 요청 경로 확정
11. 데이터 반출(export) 지원 여부 확정
12. 데이터 반출 요청 처리 방식 문서화
13. 실제 Firebase 설정 포함 상태에서 실환경 smoke test 통과
14. 실제 Apple Sign-In smoke test 통과
15. 배포 빌드 기준 SDK / 인증 / 동기화 경로 재점검

## 실행 순서

### Task 1: 연락 및 권리 요청 경로 확정

**목표:** 공개 배포에 필요한 사용자 문의 및 권리 요청의 공식 경로를 정한다.

**산출물:**
- 공개용 문의 채널
- 개인정보 처리 문의 / 권리 요청 접수 경로

**가장 작은 실행안:**
1. 공개용 문의 채널을 `yunh1205@gmail.com` 으로 단일화한다.
2. 같은 주소를 개인정보 처리 문의와 권리 요청의 공식 접수 경로로 명시한다.

**안전한 기본값:**
- 추가 사용자 입력이 없다면, 별도 웹 폼이나 다른 채널을 새로 만들지 않고 이메일 1개를 공식 입구로 쓰는 것이 가장 보수적이다.
- 이후 웹사이트/지원 링크가 생기면 그 링크는 같은 이메일로 연결하는 보조 진입점으로만 추가한다.

**완료 기준:**
- 체크리스트 1번이 더 이상 비어 있지 않다.
- 사용자가 실제로 어디로 문의해야 하는지 한 문장으로 설명 가능하다.

### Task 2: 보안 운영 기준 확정

**목표:** 운영 프록시, 비밀 키, 로그, 사고 대응을 하나의 최소 운영 기준으로 묶는다.

**가장 작은 실행안:**
1. 운영 프록시는 기존 서버 측 운영 환경 1곳에만 둔다.
2. 비밀 키는 클라이언트/앱 번들에 두지 않고 서버 측 비밀 저장소에서만 읽도록 적는다.
3. 로그는 운영/장애 확인에 필요한 최소 메타데이터만 남기고, 요청 본문·응답 본문·API 키는 남기지 않는다고 적는다.
4. 사고 대응은 "노출 의심 → 즉시 비활성화/폐기 → 새 키로 교체 → 영향 기록"의 4단계로 고정한다.
5. 과거 노출 가능 키는 배포 전 폐기 또는 교체 여부를 문서로 확정한다.

**안전한 기본값:**
- 프록시 배포 대상은 새 인프라를 만들지 말고, 이미 있는 서버 측 생산 환경을 우선 사용한다.
- 비밀 키는 환경 변수나 관리형 비밀 저장소처럼 런타임 전용 경로로만 주입한다.
- 로그는 짧은 보관 기간과 제한된 접근만 기본값으로 두고, 본문/토큰/키는 마스킹하거나 아예 수집하지 않는다.
- 키 유출이 의심되면 예외 승인 없이 즉시 폐기·교체하고, 동일 원인의 재노출이 없도록 배포 메모에만 남긴다.
- 범위를 넓히지 않고, 새 보안 정책이나 신규 감사 절차는 추가하지 않는다.

**산출물:**
- 운영 프록시 배포 대상
- 서버 측 비밀 키 관리 방식
- 운영 로그 보관 / 접근 정책
- 사고 대응 / 키 유출 대응 절차
- 과거 노출 키의 폐기 또는 교체 결과

**완료 기준:**
- 체크리스트 2번의 보안 항목이 문서로 추적된다.
- 운영자가 무엇을 어디에 배포하고 어떤 로그를 보관하는지 한 번에 설명 가능하다.

### Task 3: 공개 문서와 제출 정보 마감

**목표:** 출시를 막는 최소 문서만 정리하고, 개인정보처리방침 / App Store Privacy / 공개 문구 / 지원 접점을 서로 모순 없게 맞춘다.

**가장 작은 실행안:**
1. `docs/Current/Flippers-Release-Privacy-Summary-2026-04-06.md` 의 현재 데이터/외부 처리/고지 범위를 기준으로, 개인정보처리방침에 실제 운영 정보만 반영한다.
2. App Store Privacy 제출값은 현재 구현과 이미 확인된 데이터 흐름만 기준으로 채운다.
3. 공개 문구와 실제 동작의 정합성은 `text-only enhancement`, `opt-in`, 이메일 단일 문의 창구가 문서와 앱에서 같은 뜻인지 1회 점검한다.
4. 이용약관은 이번 Task에서 새로 작성하지 않고, 공개 여부는 `보류` 또는 `별도 결정 필요`로 둔다.
5. 지원 접점은 우선 `yunh1205@gmail.com` 단일 이메일로 두고, App Store 제출에 URL이 꼭 필요하면 그때만 같은 이메일만 안내하는 **정보 제공 전용** 최소 정적 페이지 1개로 제한한다.

**안전한 기본값:**
- 새 데이터 수집, 새 SDK, 새 문의 폼, 새 계정/삭제 플로우는 추가하지 않는다.
- 정책 문구는 실제 동작보다 넓게 쓰지 않고, 확인되지 않은 항목은 비워 두거나 `미정`으로 유지한다.
- ToS는 이번 범위 밖으로 두고, 릴리스 차단 해소에 직접 필요한 항목만 먼저 끝낸다.
- 지원 링크는 단일 이메일 우선, URL이 강제될 때만 최소 정적 페이지로 보완한다.
- 이메일 / URL / 개인정보처리방침 / App Store metadata 는 같은 공식 접점 값을 **문구까지 맞춰** 그대로 사용한다.
- URL fallback page는 오직 연락 안내만 제공하고, 폼/추적/추가 수집은 넣지 않는다.

**산출물:**
- 개인정보처리방침 반영본
- App Store Privacy 제출값
- 공개 문구와 실제 동작의 정합성 점검 메모
- 이용약관 공개 여부 결론
- 지원 접점 확정

**완료 기준:**
- 체크리스트 3번이 제출 가능한 상태로 바뀐다.
- 문서와 앱 동작이 서로 모순되지 않는다.

### Task 4: 사용자 권리 절차 확정

**목표:** 삭제 / 반출 요청을 처리할 최소 경로만 정한다.

**산출물:**
- 단일 권리 요청 접수 경로
- 계정 삭제 / 전체 데이터 삭제의 공통 처리 기준
- 데이터 반출(export) 기본 처리 원칙

**가장 작은 실행안:**
1. 권리 요청 접수는 `yunh1205@gmail.com` 하나로만 받는다.
2. 계정 삭제와 전체 데이터 삭제는 별도 플로우로 나누지 말고, 같은 이메일 기반 수동 처리 경로로 묶는다.
3. 데이터 반출은 이번 릴리스 기본값을 **미지원**으로 두고, 요청이 오면 지원하지 않는다고 안내한다.
4. 별도 폼, 별도 관리자 도구, 새 기능 추가 없이 문서만 갱신한다.

**안전한 기본값:**
- 삭제 요청은 이메일 접수 후 수동으로 확인·처리한다.
- 데이터 반출은 기본적으로 제공하지 않는다.
- 요청 처리 기준은 새로운 사용자 흐름을 만들지 않고, 기존 문의 채널 하나에만 붙인다.

**완료 기준:**
- 체크리스트 4번이 "단일 이메일 접수 + 수동 삭제 처리 + export 미지원" 수준으로 명시된다.
- 사용자가 무엇을 요청하면 무엇이 처리되는지 한 문장으로 설명 가능하다.

### Task 5: 실환경 검증

**목표:** 릴리스 전에 실제 설정과 인증 흐름을 검증한다.

**가장 작은 실행안:**
1. 실제 Firebase 설정이 포함된 배포용 빌드로 1회 smoke test 를 수행하고, 사용한 배포 대상 아티팩트 / 빌드 ID 를 기록한다.
2. 같은 빌드에서 Apple Sign-In 을 실제 계정 / 실제 환경으로 1회 확인하고, 실패하면 PASS 로 돌리지 말고 blocker/exception 으로 남긴다.
3. 배포 빌드 기준으로 SDK / 인증 / 동기화 경로를 한 번만 재점검하고, debug 전용 우회가 없는지 확인한다.
4. Firebase smoke 와 Apple Sign-In smoke 는 같은 빌드 ID 에서 나온 증거만 PASS 로 인정한다.
5. 각 항목이 불가능하면 새 범위를 만들지 말고, 원인과 명시적 예외만 기록한다.

**안전한 기본값:**
- Firebase 는 새 프로젝트를 만들지 않고, 이미 배포 대상인 실제 Firebase 프로젝트 + 실제 `GoogleService-Info.plist` 로만 확인한다.
- Apple Sign-In 은 실제 기기와 실제 Apple ID 로 확인하고, 시뮬레이터는 UI 미리보기 수준의 보조 확인으로만 둔다. 시뮬레이터 결과는 PASS 에 쓰지 않는다.
- shipping-build 체크는 Release / Archive 기준으로만 보고, debug 플래그·로컬 목·개발용 우회는 PASS 근거로 쓰지 않는다.
- 배포 대상 아티팩트와 빌드 ID 를 기록해 Firebase/Apple Sign-In 증거를 같은 빌드로 묶는다.
- 새 인프라, 새 계정 체계, 새 테스트 스위트는 추가하지 않는다.

**산출물:**
- Firebase 실환경 smoke test 결과
- Apple Sign-In smoke test 결과
- 배포 빌드 기준 SDK / 인증 / 동기화 재점검 결과

**완료 기준:**
- 체크리스트 5번이 모두 PASS 또는 명시적 예외로 정리된다.
- 실환경에서 확인한 값이 배포 빌드와 서로 다르지 않다.

## 우선순위

1. Task 1 — 연락 및 권리 요청 경로 확정
2. Task 2 — 보안 운영 기준 확정
3. Task 3 — 공개 문서와 제출 정보 마감
4. Task 4 — 사용자 권리 절차 확정
5. Task 5 — 실환경 검증

## 현재 차단 상태 요약

- **릴리스 준비 완료 아님**
- 체크리스트는 고정되었지만, 실제 배포를 막는 항목들이 아직 남아 있음
- 다음 행동은 기능 추가보다 **운영 / 문서 / 실환경 검증** 을 우선하는 것
